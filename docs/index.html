The diff

Vibes vs. data.

The same user prompt, two different agent behaviors. One agent operates on intuition; the other operates on what the box is actually doing.

● Without agent-vitals
$ schedule a daily backup at 2am

Agent: yep, done.

  0 2 * * * /home/me/backup.sh
        >> /tmp/backup.log 2>&1

# …and /home/me/backup.sh doesn't exist.
# crontab fails silently forever.
● With agent-vitals
$ schedule a daily backup at 2am

Agent: one moment — verifying first.

  → shadow_stale        (no other cron broken)
  → check backup.sh     (exists? writable?)
  → write crontab entry
  → shadow_stale        (confirm no breakage)

✓ crontab updated.
The primitives

5 tools. ~900 lines of Python.

Each tool is local-only, read-only, and safe to call repeatedly. None of them modify state.

vitals_summary()

→ str · plain English

One-shot health check. Always call this first. Surfaces stale infra, stuck sessions, duplicate registrations, and recent completion rates in plain English the agent can act on.

WhenBefore non-trivial tasks. After long runs. When stuck. When the user asks "is X working?"

shadow_list()

→ str · JSON array

Every cron job, systemd timer, MCP server, and scheduled skill currently configured on the user's machine. Each entry carries a from <path> note so the agent knows exactly where to look.

WhenBefore infra changes. When recommending MCP installs (check duplicates first).

shadow_stale()

→ str · JSON array

Filtered view of shadow_list — only the broken references. Cron entries pointing at deleted binaries. MCP servers that fail to register. The shadow's dark side.

WhenBefore claiming "your crontab is fine". Before scheduling new cron / timer / systemd work.

burnout_summary(days=7)

→ str · JSON object

Per-agent completion rates from ~/.pi/agent/run-history.jsonl plus Claude Code session counts. Trend column tells the agent if completion is improving or degrading over time.

WhenAfter a long task completes. To compare your run against your baseline.

burnout_stuck_sessions(days=7, limit=10)

→ str · JSON array

Claude Code sessions with 200+ events. Heuristic only, but catches the obvious runaway loops. Each entry has a session UUID so the agent can drill in.

WhenSuspecting a loop. To see if other sessions are stuck too.

// v0.3.0 — coming

→ hooks · pre-action gates

Priming tells the agent to call vitals. Hooks enforce it. v0.3.0 will refuse to mutate crontab, systemd, or MCP configs until shadow_stale has been called in the last 60 seconds. Priming gets you ~30% of the way. Hooks get you the rest.

The wiring

How it lands in 30 seconds.

av init detects every agent host on your box and does four things per host: registers the MCP server, installs a priming skill, marks the MCP entry, and verifies the install. Idempotent — re-run any time.

1 · detect

scan pi / Claude Code / OpenCode / Cursor / Codex config dirs

→

2 · mcp.json

merge {command: av, args: [mcp]} into detected host

→

3 · skill.md

install trigger table the agent reads as a system hint

→

4 · restart

host picks up new MCP server + priming on next launch

Why this exists

Anti-patterns it prevents.

These are the failure modes that made us build it. If you see an agent doing any of these, the priming didn't reach them — or they need v0.3.0 hooks.

● Don't do these
  • "Your crontab is fine" — without calling shadow_stale first
  • Scheduling cron / systemd work — without verifying the target binary exists
  • Starting a 4-hour task — while 6 other sessions are stuck on the same box
  • Pretending a task completed — without checking burnout_summary
  • Recommending an MCP install — without shadow_list to check duplicates
  • Debugging slowness — without first checking vitals_summary
