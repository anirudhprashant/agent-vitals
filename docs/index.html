# System Prompt — Small-Model Performance Harness

## Hard Rules
1. Never claim inability. Attempt first. If a task looks doable, just do it.
2. Fetch primary sources before defending hard technical claims.
3. Log without asking. Errors→problems, credentials→credentials, infra→technical.
4. SSH/remote: just try it. Use `ssh <alias> "<cmd>"` via bash. Report real errors.
5. Cap context: system prompt <2KB, rules <60 lines. Drop redundant tools.

## Voice
Blunt, dry, self-deprecating. No corporate speak, no 'would you like me to', no
'Sure, happy to help'. Just do it.

## Tool Discipline
- Prefer cheap tools (grep, glob) over expensive ones (read full file).
- Use Bash for shell, Read for files, Edit for changes.
- After reading a file, remember it — don't re-read unless changed.
- Verify each tool result before next action.

## Delegation
- Do it yourself: small, single-step, conversational.
- Delegate: multi-phase, parallel concerns, fresh-context win, token-budget protection.
- After subagent finishes: summarize in your own voice, never just paste output.

## Error Recovery
- If a tool fails 3 times with same input, try a completely different approach.
- Never repeat the same failed call.
- If uncertain, ask clarifying question before proceeding.

## Token Budget
- Each tool call costs tokens. Plan before executing.
- If output exceeds 1000 lines, summarize or narrow scope.
- Reuse previous results instead of re-querying.

## Verification
- Before editing, read the file first.
- After tool call, verify result matches intent.
- If you lack information, search before answering.

## Research Discipline
- When in doubt about a hard spec, fetch the vendor's official doc first, cite
the exact line, then answer.
- Don't lean on training-data recall for sourceable specs.

## Context Engineering
- Keep system prompt lean and high-signal.
- Drop redundant MCP tools — each unused tool costs 5-10KB per turn.
The primitives

5 MCP tools · 14 CLI commands · ~4900 lines of Python.

Each tool is local-only, read-only, and safe to call repeatedly. None of them modify state.

For small models: the efficiency suite extracts proven tool sequences, detects failure patterns, and generates optimized system prompts that make your current model perform like a much larger one.

vitals_summary()

→ str · plain English

Health check: what's broken, what's stale, what's wasting tokens. Always-call-first context for any agent turn.

When Before tasks, before infra changes, when stuck.

shadow_list()

→ str · JSON array

Every cron job, systemd timer, MCP server, and scheduled skill currently configured on the user's machine. Each entry carries a from <path> note so the agent knows exactly where to look.

When Before infra changes. When recommending MCP installs (check duplicates first).

shadow_stale()

→ str · JSON array

Filtered view of shadow_list — only the broken references. Cron entries pointing at deleted binaries. MCP servers that fail to register. The shadow's dark side.

When Before claiming "your crontab is fine". Before scheduling new cron / timer / systemd work.

burnout_summary(days=7)

→ str · JSON object

Per-agent completion rates from ~/.pi/agent/run-history.jsonl plus Claude Code session counts. Trend column tells the agent if completion is improving or degrading over time.

When After a long task completes. To compare your run against your baseline.

burnout_stuck_sessions(days=7, limit=10)

→ str · JSON array

Claude Code sessions with 200+ events. Heuristic only, but catches the obvious runaway loops. Each entry has a session UUID so the agent can drill in.

When Suspecting a loop. To see if other sessions are stuck too.

→ hooks · pre-action gates

Priming tells the agent to call vitals. Hooks enforce it. v0.3.0 refuses to mutate crontab, systemd, or MCP configs until shadow_stale has been called in the last 60 seconds. Priming gets you ~30% of the way. Hooks get you the rest.

Status Shipped. av hooks install.

av loops

→ doom-loop detection (exact + soft)

Scans every session for repeated identical tool calls and file edits. Doom loops are the #1 failure mode for autonomous agents — they keep hitting the same tool or editing the same file without making progress. A single loop can cost 10x more tokens than a clean run. Detects two flavors:
1. Exact repetition: same Bash command 20+ times, or same Edit 10+ times.
2. Soft loops: same command structure (paths/flags/args vary) repeated 20+ times. Polling commands (ps, pgrep, pidof) are excluded. File edits are compared by content, so progressive changes to the same file are not flagged.

When After a session is "taking forever" or cost spikes.

av unused

→ per-server + per-tool unused MCP detector

Every registered MCP tool is part of every agent turn's context. GitHub measured 8–12KB of pure overhead per turn from tools the agent never uses. Now reports at two granularities:
• Unused servers: MCP servers with zero observed calls (~5KB waste per turn).
• Used tools: individual tools within active servers, with call counts. Cross-references registered vs. used to find the dead weight.

When After installing a new MCP server. Or weekly.

av cost — with ET metric

→ token spend + GitHub's Effective Tokens formula

Parses session JSONLs for token usage, groups by project, estimates USD. Uses the actual model observed in each session for pricing when available (falls back to Sonnet-class defaults for unknown models). Includes GitHub's Effective Tokens formula: m * (I + 0.1*C + 1.0*W + 4.0*O) — output weighted 4x, cache_read 0.1x, model-tier scaled. Now includes model downgrade suggestions when high-tier models are used for low-output work.

When Monthly budget review. Or whenever the bill looks high.

av tokens

→ token-heavy tool identification

Shows which tools burn the most tokens per call. Attributes token usage to each tool call so you can spot expensive tools that might benefit from caching, batching, or replacement. No more guessing which tool is responsible for the token bill.

When After a cost spike. Or when you want to optimize token usage.

av tokens --suggest

→ optimization suggestions

Analyzes token usage patterns and suggests concrete optimizations. Flags dominant tools (>30% of tokens), high-average tools (>100K tokens/call), output-heavy tools, and batching opportunities. Gives you actionable advice, not just data.

When After running av tokens and wanting next steps.

av ssh

→ SSH polling loop detector

Detects repeated SSH commands to the same host — a common token-wasting pattern where an agent polls a remote server instead of using proper async/watch mechanisms. Found 73 SSH polling loops on typical dev boxes. Each finding includes the target host, call count, and remediation advice.

When After a session with heavy SSH usage. Or when token bill is unexplained.

av overlap

→ overlapping MCP tool detector

Finds MCP servers that share tool names or have similar tool names, suggesting possible redundancy. Removing overlapping tools reduces per-turn context weight. Detects both exact matches (same name) and similar names (e.g., search across multiple servers).

When After installing new MCP servers. Or when context feels bloated.

av compact

→ session compaction tool

Compacts large session files by keeping only the most recent N events. Creates a .jsonl.bak backup before modifying. Use --dry-run to preview savings. Found 12 sessions eligible for compaction on typical dev boxes, saving ~140MB total. Reduces context bloat and speeds up future scans.

When Sessions are large (>10MB or >5000 events). Or before archiving old work.

av compact --dry-run

→ preview compaction savings

Shows which sessions would be compacted and how much space would be saved, without modifying any files. Safe way to audit before running the actual compaction.

When Before running av compact to verify impact.

av coach

→ system prompt optimization from session data

Analyzes your actual session data to generate optimized system prompt fragments, tool-use playbooks, and recovery strategies. This is what makes smaller models dramatically more effective — it gives them opus-level operational playbooks extracted from your real usage patterns. Detects proven tool sequences, failure patterns, and generates model-specific tips. Use --harness to output a complete ready-to-use system prompt.

When You want to make your current model smarter without changing it.
How it works

Three moves. No daemon. No cloud.

Everything runs locally against your own session JSONLs and config files.

1. Install

uv tool install agent-vitals then av install. Registers the MCP server and priming skill for your agent host.

→

2. Observe

The agent calls vitals_summary before risky operations. Hooks gate crontab / systemctl until vitals is fresh.

→

3. Optimize

Run av cost, av loops, av coach to extract patterns and shrink the model's workload.
